# OpenAPI Release Artifact Specification

## Requirement: Spec generated from code, richly documented
The OpenAPI document SHALL be generated by FastAPI's native generator from the running application definition — never hand-maintained. The application SHALL supply: project title; a project-level description covering the product's purpose, the three access surfaces, and the deterministic-scoring contract; the package version (kept in lockstep with release tags by semantic-release); license and contact metadata; and the repository link in `externalDocs.url`. Every route SHALL declare a non-empty `summary`, a detailed Markdown `description`, and examples — request examples (where a body exists) and response examples — sourced from the committed eval fixtures so documentation and tests share one truth. An offline test SHALL enforce the summary/description/example coverage for every path+method.

## Requirement: Export script
`scripts/export-openapi.py` SHALL produce `openapi.json` and `openapi.yaml` by importing the FastAPI app without binding a port. While `job_matcher.api` does not yet exist, the script SHALL exit 0 with a clear "skipped" notice so the workflow can be wired ahead of the API bolt.

## Requirement: Separate release-artifact workflow
A dedicated GitHub Actions workflow (its own file, not part of release.yml) SHALL run on `release: published` and manual dispatch: check out the release tag, generate the spec, validate it with `openapi-spec-validator`, and attach `openapi.json` + `openapi.yaml` as assets of that GitHub Release. Validation failure SHALL fail the workflow and attach nothing. On non-release dispatch runs the spec SHALL be uploaded as a workflow artifact instead.

## Requirement: Standing convention
The API-documentation bar (endpoint summaries, descriptions, fixture-sourced examples; OpenAPI shipped with every release) SHALL be recorded in `AGENTS.md` as a repo-wide convention so future endpoints cannot merge undocumented.
