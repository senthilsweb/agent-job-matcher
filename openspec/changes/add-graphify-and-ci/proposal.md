# Proposal: Add graphify knowledge graph + CI automations

> Status: **APPROVED** (owner-directed, 2026-07-11) — implemented in the
> same commit; verification = first green runs of both workflows
> Owner: @senthilsweb
> Modelled on: `templrgo` (graphify scripts + workflow),
> `privacyshield` (GHCR build-and-publish workflow)

## Why

1. **AI-agent context optimisation.** As this repo grows (backend rebuild,
   frontend phase), AI coding sessions burn tokens rediscovering
   cross-references by grep. templrgo solved this with a graphify-generated
   knowledge graph regenerated by CI plus a token-cheap `graph-index.json`
   slice for LLM orientation. This repo adopts the same machinery from day
   one, while it is still cheap to wire in.
2. **Shippable from day one.** privacyshield's single GHCR workflow gives
   `docker compose up`-grade distribution. Adopting the same pipeline now
   means every push to `main` produces a pullable public image, and the
   backend rebuild inherits a working release path instead of adding one
   later.

## What changes

- **Graphify knowledge graph, dropped at the repo root** (owner decision —
  diverges from templrgo's orphan-branch storage): CI regenerates and
  commits `graph.json` (full cytoscape graph), `graph-index.json` (≤50 KB
  LLM slice), and `graph-manifest.json` (provenance) at the root of `main`.
- `scripts/graphify-build.py` — copied from templrgo (repo-generic:
  AST-only structural extraction, no LLM required).
- `scripts/graphify-index.py` — copied from templrgo, adapted: module
  roots `backend/`, `frontend/src/`, `openspec/`, `scripts/`, `.github/`;
  templrgo's RBAC cross-referencing removed.
- `.github/workflows/graphify.yml` — modelled on templrgo's: push to main
  (path-filtered) + weekly cron + manual dispatch; `[skip graphify]`
  commit-marker loop guard; failures open/update a tracking issue and
  never block `main`; optional `OPENAI_API_KEY` enables graphify's
  semantic pass (absent → free AST-only extraction).
- `.github/workflows/build-and-publish.yml` — modelled on privacyshield's:
  Buildx + GHCR publish of `ghcr.io/senthilsweb/agent-job-matcher`
  (branch/PR/semver/sha/latest tags, amd64, GHA layer cache). Auth is the
  built-in `GITHUB_TOKEN` with `packages: write` — **no repository
  secrets exist to copy from privacyshield; none are needed.**
- `Dockerfile` + `.dockerignore` at root — an interim image packaging the
  prototype CLI so the publish pipeline is green from day one; superseded
  by the `add-job-matcher-cli` rebuild (its Bolt 4 API surface becomes the
  privacyshield-style combined image in a later change).

## Secrets & visibility posture

| Item | Status |
|---|---|
| GHCR publish auth | Built-in `GITHUB_TOKEN` — nothing to configure |
| `OPENAI_API_KEY` (graphify semantic pass) | Optional. Not set: no local source held the value (templrgo's secret is write-only on GitHub). Add later via `gh secret set OPENAI_API_KEY -R senthilsweb/agent-job-matcher` — the workflow upgrades automatically |
| Public image visibility | GHCR packages are created private on first publish; flipping to public is a one-time manual step in the package settings (no API exists) |

## Out of scope

- Semantic/community enrichment of the graph in CI (needs an LLM
  orchestrator; can be run locally and pushed, per templrgo's note).
- The production combined backend+frontend image (arrives with the
  frontend change).
- Release tagging automation (templrgo's `release.yml`) — until the
  rebuild lands there is nothing meaningful to version.

## Acceptance criteria

1. Push to `main` touching `backend/`/`openspec/`/`scripts/` triggers
   graphify; the run commits refreshed `graph.json`, `graph-index.json`,
   `graph-manifest.json` at the root with a `[skip graphify]` message,
   and that commit does not retrigger the workflow.
2. `graph-index.json` stays ≤ 50 KB and groups modules by this repo's
   roots.
3. A graphify failure opens/updates a `graphify-failure` issue and does
   not block the push.
4. Push to `main` publishes `ghcr.io/senthilsweb/agent-job-matcher:latest`
   (+ sha tag); a version tag `v*` adds semver tags; PRs build without
   pushing.
5. `docker run -e API_KEY=<key> ghcr.io/senthilsweb/agent-job-matcher:latest`
   prints the prototype CLI's usage text. (Without a key it prints the
   prototype's clear "Set OPENAI_API_KEY or API_KEY" error — the
   prototype validates env before argparse; the rebuild removes this.)
   Verified locally pre-push on arm64, including the two image-level
   fixes it took: `cryptography<49` pin and `OPENSSL_armcap=0` (SIGILL
   under Docker Desktop on Apple Silicon — see Dockerfile comments).
