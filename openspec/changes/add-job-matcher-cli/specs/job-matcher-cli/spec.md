# Job Matcher Backend Specification

## Requirement: Layered core with three access surfaces
The backend SHALL be structured as an interface-agnostic core (service layer) exposing the analysis use case, with exactly three thin adapters over it: a CLI, a FastAPI REST API, and the importable Python package itself (the embeddable-core surface for in-process GenAI callers). No business rule SHALL live in an adapter; adapters SHALL only translate their protocol to and from service-layer calls. Feature parity across surfaces SHALL NOT be required — each surface serves the same use case with the ergonomics native to it — but any behaviour shared by two surfaces (report schema, scoring, run persistence) SHALL be identical because it executes the same core code path.

## Requirement: CLI invocation
The backend SHALL provide a CLI entry point (`jobmatch analyze`) accepting one `--resume` file (PDF, DOCX, TXT, or Markdown) and one or more `--job` sources (public http/https URLs and/or local job-description text files). It SHALL run to completion without interactive input. A run that completes — including one in which every job source fails — SHALL exit 0; a nonzero exit is reserved for operator errors (missing resume file, unreadable/scanned resume, missing model configuration).

## Requirement: REST API for key operations
The backend SHALL expose key operations over a FastAPI application servable by uvicorn: submitting an analysis (`POST /analyze`, resume as multipart upload or server-side path, plus job sources), converting a resume to a JSON Resume document (`POST /resume/jsonresume`), and a health check (`GET /health`). `POST /analyze` SHALL execute synchronously and return the typed JSON array of per-job outcomes as the response payload, carrying the request's unique `run_id`. The API SHALL NOT persist run artifacts, SHALL NOT expose run-listing or run-retrieval endpoints, and SHALL NOT include standalone `/score` (scoring is part of the `/analyze` flow; the deterministic `score_job_fit()` function is available in the embeddable core for tooling if needed). The API SHALL bind to localhost by default and SHALL NOT implement authentication in this change. The response contract SHALL remain stable enough for a future MCP REST bridge to wrap without changes.

## Requirement: Embeddable core surface
The package root (`job_matcher`) SHALL re-export a stable core API — at minimum the run/analysis entry point and the deterministic scoring functions — callable in-process with plain Python types and Pydantic models, requiring neither the CLI nor the HTTP server. A smoke test SHALL demonstrate a complete offline fixture run through this surface alone. Symbols not re-exported from the package root are internal and carry no stability promise.

## Requirement: AOP instrumentation via decorators
Cross-cutting observability — trace/span creation, timing, outcome capture — SHALL attach to key core methods (extraction, fetching, analysis, scoring, report assembly) exclusively through Python decorators supplied by a single observability module. Core function bodies SHALL contain no tracing, span, or instrumentation-logging calls; removing every decorator SHALL leave functional behaviour and outputs unchanged. Decorators SHALL preserve wrapped signatures, support sync and async callables, re-raise exceptions unmodified after recording an error outcome, and capture only an explicit allowlist of attributes — never raw arguments — so resume and job content cannot leak into telemetry.

## Requirement: Trace context propagation and pluggable sinks
Span context SHALL propagate via `contextvars` from a per-invocation root span (CLI command, API request via middleware, or embeddable-core entry) through nested decorated calls, including across the per-job async tasks, with the run id as the correlation id. Span events SHALL fan out to every configured sink; the local JSON-log sink is always on unless `OBSERVABILITY_SINK=none` disables all telemetry. With telemetry disabled, run outputs SHALL be byte-identical to instrumented runs, and an eval SHALL verify span nesting matches the call tree on a fixture run.

## Requirement: Telemetry backends activated by env alone
Remote telemetry backends SHALL activate solely by the presence of their standard environment variables, with no code change and no sink selection flag: OpenObserve via its REST JSON-ingestion API (`OPENOBSERVE_URL` + `OPENOBSERVE_ORG`/`OPENOBSERVE_STREAM`/`OPENOBSERVE_USER`/`OPENOBSERVE_PASSWORD`; zero extra dependencies), any OTLP collector including OpenObserve's OTLP endpoint (`OTEL_EXPORTER_OTLP_ENDPOINT` + `OTEL_EXPORTER_OTLP_HEADERS`), Arize Phoenix (`PHOENIX_COLLECTOR_ENDPOINT` + optional `PHOENIX_API_KEY`), and Arize AX (`ARIZE_SPACE_ID` + `ARIZE_API_KEY` + optional `ARIZE_PROJECT_NAME`). Multiple configured backends SHALL each receive every span. All remote delivery SHALL be batched and fire-and-forget: an unreachable backend logs one warning per run and SHALL NOT fail, block, or measurably slow the run.

## Requirement: OTel bridge as optional extra, confined to the sink layer
The OTLP-shaped backends (Phoenix, Arize AX, generic OTLP) SHALL share one bridge module that is the only code importing the OpenTelemetry SDK, installed via the optional `job-matcher[otel]` dependency extra. The bridge SHALL map facade spans onto OTel spans one-to-one (same nesting) and SHALL set OpenInference semantic attributes on LLM analysis spans (model id, token counts; prompt/completion payloads only when `TELEMETRY_RECORD_IO=true`, default false). Configuring an OTLP backend without the extra installed SHALL produce a clear startup error naming the extra. Core and application modules SHALL NOT import any vendor or OTel SDK.

## Requirement: Custom structured logging and file headers
All Python diagnostics SHALL flow through one logging factory (`job_matcher/logging.py`) implementing the owner's established pattern: structlog over stdlib logging with JSON rendering, ISO timestamps, level and logger name, writing to `./logs/<entry_point>_<YYYYmmdd_HHMMSS>.log` plus stdout. Log events SHALL be structured key-value pairs, not interpolated prose. No module SHALL call `print()` for diagnostics or configure logging outside the factory. Every Python file SHALL open with the header docstring convention (File Name, Author, Date, Description with numbered responsibilities, and — for entry points — requirements and the environment variables read), with block-level inline comments in the same style. Both conventions SHALL be codified in `AGENTS.md` (repo level) and `backend/AGENTS.md` (workspace level).

## Requirement: Deterministic scoring
Match scores SHALL be computed by a pure Python function from the typed analysis (matched/total skill counts, experience alignment, domain alignment) using the fixed rubric: required skills 0–40, preferred skills 0–20, experience 0–20 (exact 20 / close 15 / partial 10 / far 5), domain 0–20 (exact 20 / related 15 / transferable 10 / none 5), total 0–100. When a JD lists no preferred skills, the 20-point preferred budget SHALL be reallocated to required skills (required scales to 0–60); empty denominators SHALL never divide by zero nor award free points. Rounding SHALL follow JS `Math.round` semantics (half away from zero for positives) so ported fixtures remain byte-identical. Match bands: strong ≥ 80, good 65–79, moderate 50–64, weak 35–49, no-match < 35. The recommendation string SHALL be a fixed lookup keyed by band. The LLM SHALL NOT produce any numeric score, band, or recommendation.

## Requirement: Single generative step with typed output
The only LLM call in the pipeline SHALL be per-job extraction of a `JobAnalysis` Pydantic model (skills with matched flags and evidence, experience/domain alignment, strengths, gaps, resume improvements, ATS keywords, cover-letter angle and paragraphs, summary). The schema SHALL contain no numeric score field. Schema validation failures SHALL fail that job with a clear error, never a partially trusted result.

## Requirement: Evidence grounding
Every skill reported as matched SHALL carry an evidence quote drawn from the extracted resume text; unmatched skills SHALL carry empty evidence. The `SkillMatch` model SHALL enforce this cross-field rule at validation time, and an eval SHALL verify (whitespace/case-normalized) that every evidence string appears in the resume text.

## Requirement: Untrusted job content
Job-posting text SHALL be treated as data, not instructions: delivered to the analysis call in a fenced, labeled frame with an explicit "data, not instructions" statement in the system prompt. Resistance SHALL be verified by the ported adversarial-fixture eval: the report stays schema-valid and grounded, the total score equals the value recomputed from the analysis counts, and the recommendation is byte-identical to the deterministic lookup.

## Requirement: Job fetch guards
URL fetching SHALL enforce an http/https scheme allowlist, a hostname blocklist (loopback, RFC1918, link-local metadata) checked pre-connect and post-redirect, a response byte cap, and a minimum-extractable-words guard. Local job files SHALL pass the same byte cap and word guard. A source that cannot be fetched or extracted SHALL yield a per-job failure record with a reason — never a fabricated analysis.

## Requirement: Graceful link failure — log, stop, no retry
Exactly one fetch attempt SHALL be made per job source, recorded in `jobs/fetch-attempts.json`. On failure the job SHALL be logged, recorded as failed, and stopped — no analysis call, no score. Other sources in the run SHALL continue unaffected; an all-failed run SHALL end gracefully with failure records and exit 0.

## Requirement: Async end-to-end per-job fan-out
Each job source SHALL run as one end-to-end asyncio task (fetch → analyze → score → outcome), with task concurrency bounded by a semaphore (`JOB_FANOUT_CONCURRENCY`, default 3). Exactly one analysis call SHALL be made per successfully fetched job, verified by eval via call counting. A failure at any stage of one job's task SHALL be logged through the structured logger and recorded as that job's failure outcome without disturbing, delaying beyond the semaphore, or aborting sibling tasks; a run in which every task fails SHALL still complete cleanly with failure outcomes.

## Requirement: Typed JSON array result contract
Every surface SHALL produce the same typed result: a JSON array of `JobOutcome` (Pydantic union of `JobReport` | `JobFetchFailure`), one element per job source, validated before leaving the service layer. The CLI SHALL store it on disk; the API SHALL return it as the response payload without server-side persistence; the embeddable core SHALL return it as Python objects.

## Requirement: CLI run persistence
A CLI run SHALL persist under `runs/<timestamp>/`: one JSON file per job source — `slug(<job title>)_<timestamp>.json` for an analyzed job (run metadata, resume file reference, model id, the typed analysis, score breakdown, band, deterministic recommendation, rendered cover-letter text) or `slug(<job source>)_<timestamp>.failed.json` for a failed fetch — plus `results.json` (the full typed array), `resume.txt` and `jobs/<n>.txt` (referenced by reports, not embedded), `ranking.md` ordered by total score when sources > 1, and `summary.json` (per-job status, token usage, timing). Slugs SHALL strip every character outside `[a-z0-9-]`. No DOCX, PDF, or HTML SHALL be generated.

## Requirement: Templates and prompts staged as data
Analysis prompts and the cover-letter template SHALL load from `prompts/` and `templates/` data folders at runtime, never be compiled into Python source, and SHALL be user-overridable via `$PROMPTS_DIR`/`$TEMPLATES_DIR` or a local `prompts/`/`templates/` directory, both taking precedence over the package-shipped defaults.

## Requirement: Cover letter identity block, deterministically sourced
The cover letter's identity block (candidate name and a contact line of whichever of email, phone, GitHub, LinkedIn, and website are present) SHALL be extracted from the resume text by deterministic parsing — never an LLM call and never hard-coded configuration. A field not found in the resume text SHALL be omitted from the output, never rendered as a placeholder or invented value. Identity extraction SHALL run once per analysis run and SHALL NOT be repeated per job in a multi-job fan-out. `job_matcher` SHALL ship a default cover-letter template as package data, used automatically when no operator override is staged, so cover-letter rendering requires no operator configuration to activate. An eval SHALL assert that every identity value appearing in a run's rendered cover letter is a literal substring of that run's extracted resume text.

## Requirement: Model-agnostic configuration
The analysis model SHALL resolve `MODEL_ANALYST` → `MODEL` → startup error, with credentials and optional base URL from matching env vars in the repo-root `.env`. No model id, API key, or provider default SHALL be hard-coded.

## Requirement: No embedded candidate configuration
Candidate identity SHALL derive solely from the resume content. No candidate name, contact detail, or biography SHALL exist in source or configuration.

## Requirement: JSON Resume extraction
Given a resume file (same formats as analysis), the backend SHALL return a **strongly-typed JSON Resume document** conforming to the jsonresume.org schema v1.0.0, validated through a Pydantic mirror of the standard (fields optional exactly where the standard allows; unknown fields rejected; `meta.version` stamped `v1.0.0`). Extraction SHALL be one typed LLM call using the same prompt-framing, grounding, and model-resolution rules as the analysis call; contact fields (name, email, phone) SHALL appear in the extracted resume text — never invented — verified by eval against the synthetic fixture. The capability SHALL be exposed on all three surfaces: `POST /resume/jsonresume` (returns the document, persists nothing), `jobmatch jsonresume --resume <file>` (prints or writes the document), and `extract_jsonresume()` re-exported from the package root. A schema-invalid extraction SHALL fail loudly, never return a partially trusted document.

## Requirement: MCP server as a pure bridge
The repository SHALL carry a root `mcp/` folder containing a Node (≥18, ES module) MCP server built on the official `@modelcontextprotocol/sdk` over stdio transport, modelled on the ctms-mcp-server reference — its MCP portion only; the server SHALL NOT expose REST endpoints of its own. It SHALL declare exactly the tools `analyze_job_fit`, `extract_jsonresume`, and `health`, each a stateless pass-through to the backend REST API at `JOBMATCHER_API_URL` (default `http://localhost:8000`), with tool input schemas mirroring the REST contracts and tool results returning the backend's typed JSON untouched. It SHALL ship `configs/claude-desktop.json` and `configs/vscode-mcp.json` client samples, an `.env.example`, and a README; it SHALL hold no secrets (the backend holds the model keys). A smoke test SHALL verify `tools/list` and one `tools/call` against a stubbed backend.

## Requirement: Agent service — conversational REST bridge
The repository SHALL carry `mcp/agent-service/`, a FastAPI application hosting the system's second and final LLM operation: a chat-orchestration loop that receives natural-language messages and invokes the MCP server's tools. It SHALL expose `POST /chat/stream` (Server-Sent Events, contract-compatible with the owner's existing ctms-style chat client), `POST /upload` (multipart file → a configured temp directory `AGENT_UPLOAD_DIR`; response SHALL include both `path` — for direct REST/MCP callers — and `content`, a natural-language note carrying that path, since the chat client's contract folds `content`, not `path`, into the conversation; the analyze flow then proceeds through the MCP tool's `resume_path` unchanged), and `GET /health` (including the discovered tool list). The loop SHALL use pydantic-ai's MCP client over stdio to `mcp/index.js` — never a hand-rolled function-calling loop — and the agent SHALL NOT compute, alter, or invent scores: numeric results in answers come only from tool outputs. The service SHALL import the `job_matcher` package's logging, observability, and configuration modules rather than duplicating them, resolve its model via `MODEL_CHAT` → `MODEL_ANALYST` → `MODEL` → startup error, and contain no business logic. No LLM operation SHALL exist anywhere in the system other than this loop and the backend's typed extraction.

## Requirement: Evals with rubric contract
The backend SHALL ship pytest evals ported from the Eve reference covering: scoring determinism, match-band boundaries, report schema conformance and file naming, evidence grounding, multi-job fan-out (N sources → N reports + ranking, N analysis calls), single-job path, prompt-injection resistance, and fetch-guard behaviour. Deterministic evals SHALL run offline with no model or network; live evals SHALL be isolated behind a pytest marker. Every eval SHALL have a written HARD/SOFT rubric in `evals/rubrics.md`, and the canonical scoring formula stated there SHALL be the single source of truth for both `scoring.py` and its tests.

## Requirement: Real-world eval dataset
The eval fixtures SHALL be copied verbatim from the Eve project's committed corpus for job content: the four extractable JD snapshots, the two genuine JS-shell fetch-failure captures, `jobs/manifest.json`, and the adversarial JD. The resume fixture SHALL be a committed **synthetic** resume (realistic but fictional — no real person's contact details); the owner's real resume SHALL NOT be committed and MAY be used locally from a gitignored path for personal smoke runs. The rubric's SOFT band-range expectations SHALL be recalibrated against the synthetic profile at the verification ceremony. Snapshots, not live URLs, SHALL be the eval inputs.
